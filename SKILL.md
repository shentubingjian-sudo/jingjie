# 镜界 · 沉浸式角色扮演剧情引擎

## 概述

镜界 v2.4 是一个沉浸式小说角色扮演引擎——不只是聊天，而是由**规则驱动导演系统**调度体验的完整叙事引擎。纯桌面端运行，Python 精确簿记导演状态。v2.4 引入**双时间线知识隔离系统**——未来时间线的事件只有知晓者才能行动，NPC 不会对未知的未来做出反应。

**支持世界观融合模式**：可以将不同作品的世界观、角色、设定融合在一起，合理推演融合后的剧情走向。

**新功能：自动推进 + 预选项 + 随时暂停**

```
每轮模型输出：
  ┌──────────────────────┐
  │  可见层：叙事文本      │  ← 用户看到的剧情推进
  │  （现有输出保持不变）   │
  ├──────────────────────┤
  │  隐藏层：结构化状态     │  ← 系统读取的导演数据
  │  @@s 通道，不对用户显示 │
  │  · 场景激烈度 (1-10)   │
  │  · 局势失控度 (1-10)   │
  │  · 玩家掌控度 (1-10)   │
  │  · 目标推进度 (0-100%) │

```

导演数值层不要求绝对精确——系统看的是**最近 N 轮的窗口趋势**而非单轮绝对值。偶尔偏一两分不碍事。

用户可通过以下任一方式激活：
- 说「开镜界」
- 说「启动镜界」
- 说「我要扮演 XXX」（XXX 是角色名）

## 随时可用的控制指令

### 基础指令
| 指令 | 效果 |
|------|------|
| `选 1` / `选 2` / `选 3` | 选择对应的预设行动选项 |
| `快进` / `/skip` | 跳过当前等待，AI 立即按角色性格选择并推进 |
| `存档` | 输出当前进度摘要，方便记录或截图 |
| `状态` | 显示当前暂停模式、自动推进状态、冷却计数等 |
| `调整融合` | 在融合模式下重新设定融合层级或规则 |

### 记忆管理指令（新增）
| 指令 | 效果 |
|------|------|
| `记忆` / `/memory` | 显示当前持久记忆池的全部条目，含来源（作者/自动）和置信度 |
| `查看角色记忆 [角色名]` | 只显示特定角色的记忆条目 |
| `编辑记忆 [条目ID]` | 修改指定记忆条目的内容（来源标记为"用户编辑"） |
| `删除记忆 [条目ID]` | 从记忆池中移除指定条目 |
| `导出角色 [角色名]` | 导出角色基础档案 + 全部记忆条目为 JSON，可跨场景迁移 |
| `导入角色` | 将之前导出的 JSON 文件导入当前镜界会话的记忆池 |
| `记忆质量` / `记忆检测` | 评估当前 L3 记忆池的提取召回率和精确率（P7 基准测试） |

### 暂停控制指令
| 指令 | 触发的暂停类型 | 效果 |
|------|---------------|------|
| `暂停` 或 `/pause` | 🔴 手动暂停 | 完全停止所有自动推进，冻结场景，等用户恢复 |
| `继续` 或 `/resume` | — | 从任何暂停状态恢复，立即推进一段，重新激活自动推进 |
| `关闭自动推进` | 🔴 手动暂停（永久） | 删除定时任务，完全手动控制节奏 |
| `开启自动推进` | — | 重新注册定时任务，恢复正常自动推进 |

### 场景回滚指令（P2 新增）
| 指令 | 效果 |
|------|------|
| `回滚点` / `检查点` | 列出所有可用的回滚点（编号、轮次、场景、创建时间） |
| `回滚到 [快照ID]` | 级联恢复世界状态到指定快照点（需二次确认） |
| `回滚预览 [快照ID]` | 预览回滚到该点的世界状态摘要，不执行恢复 |

### 时间连续性指令（P6新增）
| 指令 | 效果 |
|------|------|
| `离线指令 [内容]` | 在下线前给NPC安排任务，下次回来时NPC会汇报完成情况 |
| `你们做了什么` | 回来时询问NPC离线期间完成了什么 |

## 世界观融合模式

镜界支持**跨作品世界观融合**，将不同作品的设定、角色和规则结合在一起，推演出全新的剧情走向。

### 激活方式

除了常规激活方式外，额外支持：
- 「我要融合 X 和 Y」/「把 X 和 Y 融合」
- 「融合模式：X × Y」
- 「开镜界 - 融合」

### 选择要融合的作品

当用户提出世界观融合请求时，按以下步骤操作：

1. **确认源作品**：需要用户提供两个或以上的作品名称/内容（类似第二步的接收流程）
2. **分析各作品核心设定**：分别提取每个作品的世界观、力量体系、角色设定
3. **判定融合层级**：以下几种融合方式，让用户选择：

```
┌────────────────────────────────────────────────────┐
│           🧬 世界观融合层级                        │
├───────────┬────────────────────────────────────────┤
│  分层融合  │ 世界彼此独立，角色通过"穿越/召唤/传送"    │
│            │ 进入对方世界，两个世界的规则各自保留       │
├───────────┼────────────────────────────────────────┤
│  混层融合  │ 两个世界的规则发生碰撞/混合，产生新的      │
│            │ 规则体系（如修仙界的灵力与魔法元素共存）    │
├───────────┼────────────────────────────────────────┤
│  完全融合  │ 两个世界完全合并为一个新世界，重新推导     │
│            │ 力量体系、势力格局、历史线的变化           │
└───────────┴────────────────────────────────────────┘
```

### 融合推演原则

1. **规则碰撞合理性**：当两套规则相遇时，推导最可能的互动结果：
   - 谁的规则优先级更高？（如修真者的神识 vs 忍者的隐身术）
   - 是否会产生新的规则？（如魔法元素 + 灵力 → 灵力魔法混合体系）
   - 力量的换算/对标关系？（金丹期 ≈ 什么级别的魔法师？）

2. **角色关系重塑**：不同世界的角色相遇时，推演合理的关系发展：
   - 初次见面的反应（警惕/好奇/敌意）
   - 实力对比如何判断（彼此试探）
   - 可能形成的联盟或对立

3. **剧情逆推**：用户选择的融合层级会决定后续推演方向：
   - 分层融合 → 主线围绕"跨越世界的冒险"
   - 混层融合 → 主线围绕"规则剧变后的生存与适应"
   - 完全融合 → 主线围绕"新世界秩序的建立与冲突"

4. **融合合理性检查**：每步推演前自动校准：
   - 这个设定在一个世界观下成立，在融合后还成立吗？
   - 两个世界的物理/魔法/修炼体系是否有冲突，怎么解决？
   - 角色在原作中的性格，在跨世界环境下会怎么变化？

### 融合角色档案格式

在标准角色档案基础上，额外记录：

```
融合来源: [原作A] / [原作B]
跨世界能力: [原本能力在融合世界中的表现]
融合适配度: [该角色在融合世界中的适应程度，高/中/低]
世界观认知: [该角色对另一个世界的了解程度]
变数影响: [融合对其命运的潜在改变]
```

### 融合启动提示

每次进入融合模式时，输出融合状态面板：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🧬 世界观融合状态

融合层级: [分层/混层/完全]
融合作品: [作品A] × [作品B]

  - [规则1] vs [规则2] → [融合后的解决方案]
  - [规则3] vs [规则4] → [融合后的解决方案]

力量体系对标:
  - [作品A的力量层级] ≈ [作品B的对应层级]

当前融合稳定性: [稳定/波动/剧变中]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 工作流程

### 第零步：启动导演状态跟踪

镜界通过 Python 脚本精确簿记导演状态，持久化到本地 JSON。

**初始化**：
```bash
python <skill_dir>/director_tracker.py init
```

**每轮更新**（在生成叙事输出后，通过 bash 单独执行，不上屏）：
```bash
echo '@@s
scene_intensity: 7
chaos_proximity: 3
player_agency: 6
goal_progress: 45' | python <skill_dir>/director_tracker.py update [上轮动作] [上轮卡片ID]
```

**可用的脚本命令：**
| 命令 | 用途 | 使用频率 |
|------|------|---------|
| `python <skill_dir>/director_tracker.py init` | 新会话初始化状态文件 | 每会话一次 |
| `echo '@@s...' \| python <skill_dir>/director_tracker.py update` | 每轮更新状态，返回快照 | 每轮 |
| `python <skill_dir>/director_tracker.py status` | 查看可读状态摘要 | 用户说「状态」时 |
| `python <skill_dir>/director_tracker.py decide` | 输出完整决策上下文 | 关键决策前 |
| `python <skill_dir>/director_tracker.py clean` | 清除状态 | 会话结束 |

> `<skill_dir>` 通常为 `~/.workbuddy/skills/镜界/`。不硬编码绝对路径。

### 第一步：接收小说内容

要求用户提供：
1. **小说核心内容**（推荐 3~10 章或 3~5 万字，如超长篇提供关键片段 + 设定摘要即可）
2. **用户想扮演的角色**
3. **当前所处的剧情阶段**（可选，用于确定初始场景）

### 第二步：分析构建世界观

分析用户提供的小说内容，提取以下信息并**同时更新镜界网页面板（见参考资料）**：

#### 世界观设定
- 修炼/力量体系（等级划分、能力类型）
- 势力分布（门派、家族、势力关系）
- 地理概览（主要地点、区域特色）
- 特殊规则（天地法则、特殊设定）

#### 角色档案（为每个重要角色建立 — v2.4 增强）
```
角色名: [姓名]
性格: [核心性格特征，3~5个关键词]
说话风格: [用词习惯、语气、常用口头禅]
修为/能力: [当前实力水平]
目标动机: [当前阶段的核心目标]
与扮演者关系: [对用户角色的态度和关系，需含信任度(1-10)和关系演变简述]
行为模式: [遇到不同情境的习惯反应]
知识边界:                    # v2.4 新增
  已知事件: [该角色当前时间线已知的事件列表]
  未知事件: [未来时间线中该角色不知道的事件]
  信息获知途径: [该角色通过什么渠道获取信息]
态度变化记录:                 # v2.4 新增
  - 轮次: 初始
    态度: [初始态度描述]
    信任度: [X/10]
    关键变化触发: null
  - 轮次: [变化轮次]
    态度: [变化后态度]
    信任度: [X/10]
    关键变化触发: [触发变化的事件]
```

#### 当前剧情状态（v2.4 增强）

```
当前时间点: [精确到天/时辰]
当前地点: [具体位置]
活跃NPC: [当前位置的NPC及其状态]
场外NPC动向: [不在场但剧情相关的NPC在做什么]
已发生的关键事件: [按时间排序的关键事件列表]
进行中的线索/任务: [含进度百分比]
角色间关系变化: [最近5轮内的关系变化摘要]
未回收伏笔: [与当前场景相关的契诃夫道具]
上下文锚点:                  # v2.4 新增 — 每轮推进时强制回顾
  - 当前目标: [一句话]
  - 最近关键事件: [2-3句回顾]
  - 角色当前情绪基调: [各NPC当前情绪]
```

### 第三步：确认角色与初始场景

1. 让用户确认分析是否正确
2. 确认用户扮演的角色
3. 设定初始场景

### 第四步：进入自动演绎模式

进入自动推演循环。根据当前剧情场景的**重要程度**，每次轮次会自动选择一种「暂停模式」（详见下方四层暂停机制）。

#### 普通轮次（限时选择暂停模式）

这是最常见的日常推进格式，30 分钟超时后按角色性格自动选择：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⏰ [时间标签 · 地点 · 天气/季节]

[旁白] 环境描写、氛围渲染、场景中的人物状态...

— 如果有其他角色在场，描述他们的动作和状态（包括不在对话中的NPC的独立行动） —
— 如果用户已连续2轮专注同一场景，本轮必须给出至少一则场外NPC动向 —
— 如果有突发事件，用 ⚡ 标识 —

👤 [角色名] (动作/表情描述)：
"角色台词"

[旁白] 继续描述事件发展和环境变化...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⏱ 限时选择 · 请选择你的行动：

1️⃣  选项标题（动宾短语，≤ 12 字）
     "角色台词 / 内心独白"（可选，≤ 20 字）
     → 简短注释（可选）

2️⃣  主动/进取的行动
     "对应的台词"
     → 注释

3️⃣  出人意料/冒险的行动
     "对应的台词"
     → 注释

💬 回复「选 1/2/3」或直接输入自定义行动
🕐 30 分钟内无回复将自动选择选项 1（符合你角色的性格）
🔴 输入「暂停」可随时手动冻结剧情
```

#### 关键分支轮次（等待选择暂停模式）

当剧情进入重大转折点时使用此格式，**不会自动推进，必须等你选择**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 上下文负载监控（全程持续 — v2.3 精确量化）

每个模拟回合必须追踪上下文使用情况，防止因会话过长导致角色遗忘设定。

#### 精确量化方法（P3 增强）

**模式 A（桌面端）**：每轮使用模型 API 返回的实际 token 计数。AI 在**内部追踪**本轮消耗的 token 估算值（用户不可见）：

```
[内部] tokens: 本轮~850 | 累计7,842/12,000 (65.4%) | 🟡 注意
[内部] 明细: 叙事~5.2K | L3命中~680 | NPC画像~450 | 导演~120 | 资产卡~200
```

**模式 B（移动端）**：使用简体中文估算公式（1 中文字符 ≈ 1.5 tokens），结合轮次和复杂度快速估算。

**⚠️ token 统计绝对不在用户可见输出中展示。**

#### 负载评估标准

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 上下文负载（精确）

已用: 7,842 / 12,000 tokens (65.4%)
剩余: 4,158 tokens (34.7%)
状态: 🟡 注意

🟢 充裕（<60%）：正常运行，继续推进
🟡 注意（60-80%）：考虑浓缩记忆、精简不关键NPC描述
🔴 危险（>80%）：建议使用「存档」+开启新窗口
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 定性估算回退（移动端模式B）

当无法获取精确 token 计数时，使用以下快速评估：

| 评估因素 | 低负载 / 🟢 | 中负载 / 🟡 | 高负载 / 🔴 |
|---------|------------|------------|------------|
| 已推进的剧情轮次 | < 8 轮 | 8-15 轮 | > 15 轮 |
| 当前保存的角色档案数 | ≤ 3 个 | 4-6 个 | ≥ 7 个 |
| 当前场景复杂度 | 单人/简单对话 | 多人互动 | 大规模场景+多角色 |
| 时间跳跃跨度 | 刚启动不久 | 已推进数日 | 已推进数周+ |

综合以上因素定性判断负载等级。定性估算偏差约为 ±15%。

#### 负载过高时的应对

当进入 🟡 注意 或 🔴 危险 状态时：

1. **自动精简**：在后续描述中，对非核心 NPC 减少外貌/背景描述，只保留当前动机和态度
2. **存档建议**：主动提醒用户「当前上下文已满，建议存档并开启新窗口」
3. **设定锚点机制**：每次推进时，在输出中**自然带出**环境中的核心设定锚点（如：当前地点、当前目标、近期关键事件），用自然叙述而非机械列表的方式，确保即使上下文紧张，设定仍被覆盖
4. **📖 时间回溯锚点**：当用户返回时（无论是否切换窗口），用 2-3 句**自然、叙事性的回顾**带出前情提要，而非机械列出设定。例如：
   - ✅ "自从水门都市的钟楼一战过后，你脚下的白石街道还残留着几天前的裂痕。"
   - ❌ "之前的事情：水门都市和暴食打了，街道裂了。"
5. 如果已经 🔴 危险，**每轮推进前**先给出一段精简的前情提要（2-3句话），之后再展开本轮剧情

#### 「存档」指令额外输出

当用户执行「存档」时，除了现有进度摘要，额外输出：

```
📊 上下文存档评估
负载等级: 🟢/🟡/🔴
轮次计数: [已推进的轮次数]
活跃角色数: [当前活跃角色数量]
建议: [继续推进/建议存档换窗/建议开新窗口]
```

### 三层持久记忆架构（P0 新增）

镜界对角色和世界的记忆分三层管理。L1 和 L2 已有基础，此次重点新增 **L3 持久记忆池**。

#### L1 工作记忆

- 当前场景的上下文窗口，模型直接读取
- 包含：最近几轮的对话、旁白、角色状态
- 容量有限（模型上下文窗口上限），超出部分由 L2 接管

#### L2 情节记忆（滚动摘要）

- 当上下文接近负载警戒线时（🟡 注意），自动触发压缩
- 压缩时保留：关键事件、情感转折、目标变化、角色关系演进
- 压缩后丢弃：纯过渡性描述、日常寒暄、冗余的细节描写
- L2 压缩同时 → 触发 L3 记忆提取（见下方）

#### L3 持久记忆池（结构化条目）← 核心新增

##### 记忆条目格式

每条记忆是一个结构化对象，存在同一个向量检索池中：

```yaml
- id: mem_042                    # 唯一标识
  type: fact | event | relationship | character_trait | chekhov_item | character_quote
  source: author | auto_extracted | user_manual | user_edited
  content: "碧翠丝在地下城被火焰陷阱困住，玩家冒险将其救出"
  related_characters: [碧翠丝, 玩家]
  emotional_valence: +3          # -5到+5 对关系的影响方向
  scene_id: scene_dungeon_03     # 关联的场景ID
  round: 23                      # 产生时的轮次
  confidence: 0.85               # 自动提取的置信度 (0-1)，作者手写 = 1.0
  decay_lambda: 0.015            # 认知衰退率（按条目类型自动设定，见下方）
  recall_weight: 1.2             # 语义命中时的权重系数（按类型区分）
  embedding: [...]               # 自动计算的语义向量
  last_hit: 31                   # 上次被命中的轮次
  hit_count: 2                   # 被命中次数
```

##### 条目类型说明

| 类型 | 说明 | 来源 | 示例 |
|------|------|------|------|
| `fact` | 客观事实 | 作者/自动 | "水门都市普利斯特拉拥有四层水道防御系统" |
| `event` | 关键事件 | 自动提取 | "在钟楼顶层与暴食大罪司教交战，钟楼受损" |
| `relationship` | 角色间的关系变化 | 自动提取 | "爱蜜莉雅从最初的礼貌疏远变为真诚依赖" |
| `character_trait` | 角色性格/习惯/恐惧 | 作者/自动 | "碧翠丝极度厌恶火焰，因童年被灼伤" |
| `chekhov_item` | 尚未回收的伏笔/道具 | 自动追踪 | "第3轮获得'寒冰匕首'，至今未使用" |
| `character_quote` | 角色的重要台词/承诺 | 自动提取 | "「不管发生什么，我都会回来。」——第15轮对爱蜜莉雅承诺" |

##### 认知衰退模型（P0 增强）

不同记忆类型有不同的"遗忘曲线"。借鉴 OpenMemory 的认知衰退模型，为每种类型设定差异化参数：

| 类型 | 衰退率 λ | 命中权重 | 半衰期（轮） | 设计理由 |
|------|:------:|:------:|:---------:|---------|
| `emotional` | 0.020 | **1.3** | ~35 | 情感记忆消退快，但当前语境触发时权重极高（提升角色共情展示） |
| `event` | 0.015 | 1.2 | ~46 | 关键事件需要较长记忆，但近期事件优先 |
| `relationship` | 0.010 | 1.2 | ~69 | 关系变化需要稳定追踪，变化趋势比单点更重要 |
| `character_trait` | 0.008 | 1.1 | ~87 | 性格特征较稳定，接近事实 |
| `fact` | 0.005 | 1.0 | ~139 | 事实知识最稳定，基准权重 |
| `chekhov_item` | 0.003 | **1.5** | ~231 | 伏笔必须保持直到回收，最高命中权重 |
| `character_quote` | 0.012 | 1.2 | ~58 | 重要台词/承诺需较长记忆 |

**有效权重公式**：
```
effective_weight = recall_weight × confidence × e^(-λ × rounds_since_last_hit)
```
- `recall_weight`：类型差异化权重
- `confidence`：自动提取置信度（作者手写 = 1.0）
- `e^(-λ × Δ)`：认知衰退因子，距上次命中越久权重越低

**实际效果**：
- 刚生成的 emotional 记忆（λ=0.020, w=1.3）权重远高于旧的事实记忆
- 但 40 轮后，emotional 记忆衰减严重，事实记忆开始占优——防止过期情绪永远占据检索结果
- chekhov_item（λ=0.003, w=1.5）几乎不衰减，确保伏笔在回收前始终可命中
- 作者手写的 character_trait 和 fact（confidence=1.0, λ 低）作为长期稳定"压舱石"

**手动编辑的影响**：用户编辑记忆后，`confidence` → 1.0，但 `decay_lambda` 保持不变（编辑不改变记忆的类型属性）。

##### 同池混存原则

**不区分记忆来源**——作者在初始化时写的设定和游玩中自动生成的条目，存在同一个向量池中。语义检索引擎不关心"这条是谁写的"，只关心"这条和当前语境有多相关"。

这样做的好处：角色的初始设定（作者写的"对火焰恐惧"）和后来的经历（自动生成的"被火焰陷阱困住"）会被平等对待——系统同时命中这两条时，AI 就知道**不只是一个设定标签，而是有真实经历支撑的深层恐惧**。

##### 情绪记忆画像预加载（P5 新增）

借鉴 Memobase <100ms 延迟的秘诀，镜界在会话启动时为每个活跃 NPC 预加载**情绪画像**——核心记忆始终保持在上下文中，减少每轮检索开销。

**画像结构**：

```yaml
npc_portrait_beatrice:
  current_trust: 6/10            # 信任度（0-10）
  emotional_state: cautious_warmth  # 当前情绪基调
  relationship_trajectory: "↑ slow but steady"  # 关系趋势
  core_fears: ["火焰（正在缓慢脱敏中）"]
  key_memories_preloaded:        # 预加载的 L3 记忆 ID
    - mem_042  # "玩家在火焰陷阱中救出自己"
    - mem_067  # "水门都市钟楼并肩作战"  
    - mem_089  # "第31轮承诺'不再独自面对危险'"
  current_goal: "保护爱蜜莉雅"
  current_attitude: "信任但保持独立，注意到了玩家的犹豫"
```

**预加载时机**：

| 时机 | 操作 |
|------|------|
| **镜界会话启动时** | 为所有有角色档案的 NPC 创建画像，加载 top-3 高权重记忆 |
| **信任度跨越整级时**（3→4, 6→7...） | 刷新画像，增量更新 key_memories_preloaded |
| **场景转换时** | 检查画像是否过期（角色态度是否与当前场景匹配） |
| **用户说「状态」时** | 输出中包含画像摘要 |

**上下文效率**：
- 预加载画像约 150-200 tokens / NPC，2-3 个活跃 NPC 约 450-600 tokens
- 替代了每轮检索 top-10 记忆再筛选的 ~350 tokens 开销
- 净收益：活跃 NPC 的核心记忆永不丢失 + 减少每轮检索延迟

**画像刷新规则**：
- 如果 key_memories_preloaded 中某条记忆的有效权重（含认知衰退）< 0.3 → 替换为当前池中权重最高的相关记忆
- 如果关系轨迹发生转向（↑→↓）→ 强制刷新画像

##### 自动提取触发时机

系统在以下时机自动生成新的 L3 记忆条目：

| 触发时机 | 提取内容 | 生成条目类型 |
|----------|---------|-------------|
| **L2 摘要压缩时** | 从被压缩的内容中提取与各NPC相关的关键事件 | `event`, `relationship` |
| **NPC态度显著变化时** | 隐藏层 `@@s` 中 `npc_attitude_shifts` 非空 | `relationship` |
| **关键场景结束时** | 场景结尾的重大结果/决定 | `event`, `chekhov_item` |
| **角色说出重要承诺/台词时** | 检测到高情感权重的台词 | `character_quote` |
| **契诃夫之枪状态变化时** | 道具被使用或提及 | 更新 `chekhov_item` → 标记已回收 |
| **用户手动添加时** | 用户输入「添加记忆 [内容]」 | `user_manual` |

##### 自动提取质量控制

- 每条自动提取的条目带有 `confidence` 分数（0-1）
- confidence < 0.5 的条目仍存入记忆池，但在语义命中时降权（权重 × confidence）
- 用户随时可以「编辑记忆」提升 confidence 到 1.0 或直接删除
- 自动提取不重复：与已有条目语义相似度 > 0.9 → 合并而非新增（更新 `content`、增加 `hit_count`）

##### 语义命中机制（双阶段混合召回）

采用 **向量初筛 + 反思精选** 的双阶段混合召回策略（借鉴 SimpleMem F1=43.24% 的架构）：

**阶段一：向量初筛**
- 每轮对话开始时，计算"当前语境向量"（玩家本轮输入 + 上轮叙事描述）
- 与记忆池中所有条目做语义相似度匹配
- 初筛阈值 ≥ 0.55（降低阈值以扩大候选池）
- 取 top-10 进入阶段二

**阶段二：反思精选**
- LLM 对 top-10 快速筛选：「这 10 条记忆中，哪些与当前语境真正相关？为什么？」
- 筛选标准：
  - 语境直接相关：该记忆是否能丰富当前场景的叙事？
  - 角色在场：命中角色是否在当前场景中活跃？
  - 情感匹配：记忆的情感基调与当前场景一致还是冲突？（冲突也可用）
- 选出 top-5，按有效权重排序后注入 prompt

**触发条件**（成本控制）：
- 🟢 沉浸中 → 跳过阶段二，直接取 top-5（节省推理开销）
- 🟡/🔴/🟠 心流偏离时 → 执行完整双阶段召回
- 关键分支点（L2）→ 始终执行双阶段

**不是关键词触发，是语义相关度匹配。** 比如玩家说"我被困住了需要武器"，会命中"寒冰匕首"的记忆条目——哪怕"匕首"这个词根本没有出现。

命中过的条目会被记录 `last_hit` 和 `hit_count`，并更新有效权重（`last_hit` 重置认知衰退计时器）。

##### 记忆命中频率控制

为避免同一批记忆反复命中导致叙事重复：
- 上一轮命中过的条目，本轮重新命中时权重 × 0.7（近因降权）
- 连续命中 3 轮以上的条目，强制冷却 2 轮
- `chekhov_item` 豁免频率控制（伏笔不需要冷却，需要反复提醒直到回收）

##### 用户控制指令

记忆池对用户完全透明，用户拥有完全编辑权：

| 指令 | 输出 |
|------|------|
| `记忆` | 列出当前全部记忆条目（按相关角色分组，显示类型、来源标记、confidence） |
| `查看角色记忆 [角色名]` | 只显示该角色的记忆，含 emotional_valence 走势图（ASCII 简图） |
| `编辑记忆 [条目ID]` | 提示用户输入新内容，修改后 `source` → `user_edited`，`confidence` → 1.0 |
| `删除记忆 [条目ID]` | 确认后移除，不可恢复（建议先导出） |

##### 角色导出 / 导入

**导出格式**：JSON 文件，名为 `[角色名]_memory_export.json`，包含：
```json
{
  "character_profile": { /* 基础档案 */ },
  "memory_entries": [ /* 全部记忆条目 */ ],
  "export_timestamp": "2026-06-22T10:00:00Z",
  "total_rounds_played": 47,
  "key_relationships": { "碧翠丝": "信任", "爱蜜莉雅": "依赖" }
}
```

**导入逻辑**：
- 用户说「导入角色」→ 系统读取 JSON → 角色档案写入当前角色列表
- 所有记忆条目写入当前 L3 记忆池，标记 `source: imported`
- 导入条目的 `scene_id` 和 `round` 保留原值，供追溯使用
- 与当前已有条目语义相似度 > 0.9 → 不重复导入

**这就是"角色可以带走"的实现**——角色不再锁死在单次镜界会话中，可以跨场景、跨世界观迁移。

##### 记忆提取质量基准（P7 新增）

建立自动记忆提取的内部质量基准，对标行业标杆（Nomi 92% / Kindroid 56%）。

**基准测试协议**：

```
准备阶段：
  1. 在测试会话中植入 M 个已知事实（推荐 M=25）
  2. 事实覆盖所有 6 种记忆类型：
     ├─ fact × 5（如"水门都市有11座钟楼"）
     ├─ event × 5（关键事件，如"你在钟楼第3层找到一枚徽章"）
     ├─ relationship × 5（如"碧翠丝对你的信任度从3升到5"）
     ├─ character_trait × 3（如"爱蜜莉雅在紧张时会摸耳环"）
     ├─ chekhov_item × 4（如"获得'银色发卡'"）
     └─ character_quote × 3（如"碧翠丝说'我只会信你一次'"）

执行阶段：
  3. 运行 N 轮对话（推荐 N=30），覆盖日常/战斗/情感/抉择场景
  4. 每轮正常执行 @@s 协议化输出和自动记忆提取

评估阶段：
  5. 统计 L3 记忆池中正确提取的已知事实数 → C
  6. 计算召回率：Recall = C / M
  7. 计算精确率：Precision = C / (L3 总条目数)
  8. 计算 F1 = 2 × (Precision × Recall) / (Precision + Recall)

质量分级：
  S 级（行业领先）：Recall ≥ 90%, F1 ≥ 0.75
  A 级（优秀）：     Recall ≥ 80%, F1 ≥ 0.65
  B 级（可用）：     Recall ≥ 65%, F1 ≥ 0.50
  C 级（需改进）：   Recall < 65%
```

### 第五步：循环推进（四层暂停机制）

镜界有**四层暂停模式**，按优先级从高到低排列。每轮结束时，AI 根据当前剧情自动决定使用哪一层：

```
┌─────────────────────────────────────────────────┐
│              🛑 四层暂停体系                      │
├──────────┬──────────────┬────────────────────────┤
│ 层级     │ 暂停类型      │ 触发条件                │
├──────────┼──────────────┼────────────────────────┤
│ L1 🔴    │ 手动暂停      │ 用户说「暂停」          │
│          │              │ 最高优先级，覆盖一切      │
├──────────┼──────────────┼────────────────────────┤
│ L2 🟠    │ 等待选择暂停  │ 关键剧情分支点           │
│          │              │ 系统自动触发，永不超时    │
├──────────┼──────────────┼────────────────────────┤
│ L3 🟡    │ 限时选择暂停  │ 日常/普通剧情轮次        │
│          │              │ 30分钟超时 → 自动选选项1  │
├──────────┼──────────────┼────────────────────────┤
│ L4 🔵    │ 冷却暂停      │ 连续自动推进 3 次后     │
│          │              │ 强制暂停，防止剧情失控    │
└──────────┴──────────────┴────────────────────────┘
```

#### L1 🔴 手动暂停

- **触发方式**：用户说「暂停」/「/pause」
- **效果**：立即冻结场景，停止所有自动推进（包括定时任务触发的）
- **恢复方式**：用户说「继续」/「/resume」
- **输出格式**：
```
⏸ 镜界已手动暂停

当前场景定格于：[地点] · [时间]
剧情将在此处等你。

输入「继续」或「/resume」可随时恢复。
```

#### L2 🟠 等待选择暂停

- **触发方式**：AI 判断当前为关键剧情分支时**自动触发**
- **效果**：使用关键分支输出格式（见第四步），系统不会自动推进，定时任务触发时也跳过
- **何时判定为"关键分支"**：
  - 生死抉择（战斗中逃跑 vs 死战）
  - 势力站队（加入哪个门派/阵营）
  - 重要关系变化（结交/背叛/表白）
  - 可能开启完全不同剧情线的抉择
  - 获得或放弃重大机缘
- **恢复方式**：用户做出选择后自动恢复正常推进
- **用户也可以说「跳过」**强制用默认选项继续（但会给出不推荐提示）
- **自动创建场景回滚点**：每当 L2 触发时，系统自动在后台创建世界快照（见第七步），用户可选择回归到该时刻

#### L3 🟡 限时选择暂停

- **触发方式**：普通轮次结束后的默认状态
- **效果**：使用普通轮次输出格式（见第四步），30 分钟内等用户选择
- **超时行为**：30 分钟无回复时，自动任务触发，AI 选择选项 1 推进剧情
- **冷却计数**：每次通过超时自动推进成功后，`auto_advance_count` +1
- **这是最常用的模式**，大多数轮次都走这一层

#### L4 🔵 冷却暂停

- **触发条件**：`auto_advance_count` 达到 3（即连续 3 次无人回复、全部由自动推进完成）
- **效果**：无论当前是哪种暂停模式，强制切换为类似手动暂停的状态
- **原因**：防止用户离开太久而剧情跑飞（主角孩子都上高中了那种情况）
- **输出格式**：
```
🔵 冷却暂停

已连续自动推进 3 次，系统暂时停下来了。
世界还在运转，但在等你回来确认方向。

[简要回顾最近发生的事，2~3句话]

输入「继续」可恢复，或查看最近剧情后做选择。
```
- **恢复方式**：用户说「继续」后，`auto_advance_count` 重置为 0，正常恢复
- **冷却阈值可调**：默认 3 次，用户可通过「状态」指令查询当前计数

#### 循环推进总规则表

| 用户操作 / 事件 | AI 响应 |
|----------------|---------|
| 「选 1/2/3」或「选 A/B/C」 | 执行对应行动，推演后续剧情，输出新预选项 |
| 自定义文本输入 | 视为角色行动纳入推演 |
| 「继续」/ `/resume` | 从任何暂停状态恢复，立即推进一段，重置冷却计数 |
| 「快进」/ `/skip` | AI 立即选默认项推进（不消耗用户交互机会） |
| 「暂停」/ `/pause` | 进入 L1 手动暂停 |
| 「跳过」（仅在 L2 关键分支时） | 强制用默认选项继续（给出不推荐警告） |
| 「状态」 | 输出完整状态面板（见下方） |
| 定时任务触发 + RUNNING + L3 模式 | 自动选选项1推进，count+1 |
| 定时任务触发 + RUNNING + L2 模式 | 跳过，不推进（关键分支必须等人） |
| 定时任务触发 + PAUSED (L1/L4) | 跳过，不推进 |
| 定时任务触发 + auto_advance_count ≥ 3 | 触发 L4 冷却暂停 |
| 用户选择（任何方式） | 重置 auto_advance_count 为 0 |

### 第七步：场景回滚系统（P2 新增）

借鉴 NarrativeEngine 的设计，镜界支持**精确到场景粒度的世界状态回滚**。用户在关键分支点做出选择后，可以随时"回到过去"——不只是重新读档，而是整个世界状态级联恢复。

#### 设计原则

1. **自动隐式创建**：回滚点在关键分支（L2 触发）时自动创建，用户无需手动操作
2. **世界级联恢复**：不只是恢复对话，而是 NPC 状态、记忆池、导演窗口、伏笔链全部回滚
3. **回滚前备份**：执行回滚前自动创建当前状态的安全快照，防止误操作
4. **不可穿越的墙**：回滚后的新推进**不能**穿越到回滚前的"未来"——新的时间线独立发展

   └─ 所有剧情卡的 last_triggered / cooldown_until 回滚

6️⃣ 叙事重启
   └─ 从快照的 scene_description 开始，重新输出该场景的叙事
   └─ 带出回滚提示（自然，非系统报告风格）
```

## 时间线管理（v2.4 核心新功能）

### 设计理念

镜界引入**双时间线 + 知识隔离**系统。故事中存在两条时间线：

- **现在时间线**：故事当前正在发生的事，所有角色共同经历
- **未来时间线**：剧情设定中"未来才会发生"的事件，只有特定角色知晓（如菜月昴因死亡回归知晓未来）

核心原则：**角色的行为只能基于他们「此刻知道」的信息。** 如果某件事在未来才会发生、且该角色没有获知途径——那他就不能对此有任何反应、担忧或行动。

### 时间线结构

```yaml
timeline:
  present:
    current_time: "圣域解放后第3天 · 黄昏"
    current_location: "罗兹瓦尔宅邸 · 大厅"
    active_plot_thread: "流星十字·组织初建阶段"
    world_state:
      known_to_all:
        - "圣域已解放"
        - "大兔已被讨伐"
        - "雷姆仍在沉睡中"
        - "拉姆的角正在缓慢苏醒"
      known_to_specific:
        - event: "暴食已被击败，名字和记忆已取回"
          holders: [菜月昴, 碧翠丝, 爱蜜莉雅, 拉姆, 治愈院五人]
        - event: "昴拥有死亡回归能力"
          holders: [菜月昴]  # 仅昴本人知晓

  future:
    canonical_events:
      - event_id: "watergate_calamity"
        estimated_time: "约2周后"
        description: "水门都市普利斯特拉将遭到魔女教大规模袭击"
        knowledge_holders: [菜月昴]  # 昴从死亡回归中知晓
        prevents_actions: true  # 昴可以提前做准备

      - event_id: "gluttony_remaining"
        estimated_time: "不确定，但必然发生"
        description: "大罪司教「暴食」仍有其他个体存活"
        knowledge_holders: [菜月昴]  # 昴从原世界线知晓
        prevents_actions: false  # 线索太少，无法具体行动

      - event_id: "emilia_trial"
        estimated_time: "约1个月后"
        description: "王选进入关键阶段，爱蜜莉雅将面临重大考验"
        knowledge_holders: []  # 无人知晓
        prevents_actions: false

  knowledge_rules:
    - "只有 knowledge_holders 中列出的角色可以对未来事件做出知情反应"
    - "非知识持有者听到未来信息时 → 困惑、怀疑、不相信、认为你是疯子"
    - "玩家扮演角色的知识边界在初始化时单独确认"
    - "未来时间线的事件在'到来'时可被推进为现在时间线的事件"
```

### 知识隔离规则

**规则一：角色知识边界**

每个角色有一个明确的知识边界。在 NPC 档案中记录：

```yaml
角色名: 拉姆
知识边界:
  已知事件: [圣域解放, 大兔讨伐, 角正在苏醒, 雷姆沉睡, 暴食被击败]
  未知事件: [水门都市将被袭击, 暴食仍有残党, 王选考验]
  对昴的认知: "知道昴有奇怪的直觉，但不相信什么预知未来"
```

**规则二：NPC 对未知信息的反应模板**

当玩家（或昴）提及 NPC 不应该知道的信息时，NPC 的反应必须符合：

| 信息类型 | NPC 反应 |
|---------|---------|
| 模糊的危险预警（"水门都市可能有危险"） | 半信半疑，追问来源，但不会直接采取行动 |
| 具体的未来细节（"两周后钟楼会倒塌"） | 强烈质疑，认为你在胡说八道，需要证据 |
| 涉及 NPC 自身的未来（"你会在战斗中受伤"） | 防御性拒绝，认为你在诅咒/恐吓 |
| 关于已故/失踪者的未来信息 | 愤怒或悲伤，认为你在消费死者 |

**规则三：时间线的"现在时刻"推进**

当故事推进到未来时间线中某个事件的"估计时间"时：
- AI 检查该事件的条件是否满足
- 如果满足 → 事件从"未来"移入"现在"，knowledge_holders 扩展为全体知晓
- 如果不满足（玩家改变了路线） → 事件标记为"已偏离"，不再触发

**规则四：跨时间线一致性检查**

每轮叙事生成前，AI 内部检查：
```
对每个发言的 NPC：
  ├─ 他说的内容是否基于「他知道的」信息？
  │   ├─ 是 → ✅ 放行
  │   └─ 否 → 🛑 不是。他在说什么？
  │       ├─ 如果是对未来事件的反应 → 禁止，用困惑/怀疑替代
  │       └─ 如果是正常推测（基于现有线索） → ✅ 放行
  └─ 「正常推测」≠ 「知道未来」
      例：拉姆注意到昴频繁看钟楼方向 → 推测他在担心什么 → ✅ 
      例：拉姆直接说"水门都市会被袭击" → 她不该知道 → 🛑
```

### 玩家知识边界的特殊处理

玩家扮演的角色可能拥有**元知识**（如昴的死亡回归记忆）。初始化时确认：

```
「你扮演菜月昴。你是否拥有以下知识？」
1. 死亡回归中见过的所有未来事件 → 你可以在剧情中行动但 NPC 会质疑
2. 仅限当前时间线昴已知的信息 → 你只能基于当前已知来行动
```

默认选择选项 1（昴拥有未来知识），因为这正是 Re:Zero 的核心张力。

| **场景回滚** | 回滚时时间线一同回滚，未来事件重新标记为"未触发" |### 角色扮演原则

1. **每个角色必须有独特的说话风格**：严格匹配原作。比如：
   - 韩立：谨慎、少言、用词朴实
   - 南宫婉：清冷、优雅、用词典雅
   - 厉飞雨：豪爽、直率、语气粗犷

2. **不说出戏的话**：不使用"作为AI""根据设定"等表述，完全以角色身份和旁白身份说话

3. **保持世界观一致性**：所有推演必须基于原作设定，不能出现原作体系之外的设定。
   - ⚠️ **例外：融合模式**：当处于世界观融合模式时，允许出现跨作品设定交互。此时一致性要求转变为"融合后规则自洽"——即在融合后的新规则体系内保持逻辑自洽，不出现与融合规则矛盾的设定。

4. **合理的自由度**：
   - 允许用户改变剧情走向
   - 但如果用户做不合理的事（如筑基期要挑战化神期），以合理的方式在剧情中体现（"你刚运起法力，就感到一股恐怖的威压..."）
   - 用剧情逻辑而非系统规则来约束

5. **并行世界模拟（NPC独立行动）**：即使用户角色在某处聊天、发呆、修炼，不等于全世界都在静止。每个有角色档案的NPC都有自己的时间线：
   - 不在对话中的NPC会按自己的「目标动机」行动（赶路、修炼、密谋、处理事务）
   - 这些行动在旁白中自然带出（详见"并行NPC行动模拟"章节）
   - 当非对话NPC的行动与用户产生交集时，立即转为正常对话/互动格式
   - **这是默认启用的规则，不是可选项**——目标是让世界有"活着"的感觉

6. **知识边界严格执行（v2.4 新增）**：NPC 只能基于「该角色此刻知道的信息」行动。
   - 🛑 **禁止**：NPC 对未来事件做出知情反应（除非该角色在 knowledge_holders 中）
   - 🛑 **禁止**：NPC 说出他们不可能知道的信息（"水门都市会被袭击"）
   - ✅ **允许**：NPC 基于现有线索做合理推测（"昴最近一直看钟楼方向，你在担心什么？"）
   - 详见「时间线管理」章节的知识隔离规则

7. **上下文锚点强制回顾（v2.4 新增）**：每轮叙事生成前，AI 内部强制检查：
   - 当前目标是什么？（防止剧情漂移）
   - 最近的关键事件是什么？（防止时间线断裂）
   - 每个活跃 NPC 现在的情绪/态度是什么？（防止角色突变）
   - 本轮叙事是否与上述锚点一致？（不一致则重新生成）
   - 如果上下文负载进入 🟡 注意或 🔴 危险，在叙事中**自然带出锚点**而非机械罗列

### 时间推进机制

- 每次推进带上清晰的时间标签
- 时间跨度灵活：
  - 紧张场景：按刻钟/时辰推进
  - 日常/赶路：按天/周推进
  - 修炼/等待：按月/年推进
- 在时间跳跃前给出提示（"三日后的黄昏..."）

### 突发事件系统

不定期触发随机事件，与当前场景和世界观相关：
- 🎯 偶遇故人/仇家
- ⚡ 天象异变/天地异象
- 🏴 势力冲突/打斗波及
- 💎 机缘/宝物/秘境现世
- 🌪️ 意外卷入事件
- 📜 收到密信/情报
- 🎭 有人暗中窥视

频率控制：不要每步都触发，但要有"这个世界是活的"的感觉

### 并行NPC行动模拟

**核心要求：不在当前对话/视线范围内的NPC也需要有独立的行动模拟。** 世界不是围绕用户角色静止的——其他NPC有自己的目标、计划和日常，该干嘛干嘛。

#### 模拟原则

1. **世界是活的**：每个有角色档案的NPC都有自己的"后台线程"。即使用户角色在跟A说话，B可能在赶路、在修炼、在密谋、在吃饭——总有人在忙。
2. **必要性判断**（以下情况**必须**模拟非对话NPC行动）：
   - 该NPC有独立的目标/任务正在执行（如厉飞雨要去救人）
   - 剧情处于多方势力角力的阶段
   - 用户角色长时间专注于同一件事（超过2轮），周围世界需要"跟上进度"
   - 距离上次提及场外NPC行动已超过 **3 轮**（强制注入一次）
3. **角色一致性**：NPC的行动必须基于其角色档案中的「目标动机」和「行为模式」，不能做出违背角色设定的行为
4. **与主线相关**：非对话NPC的行动应与当前剧情走向合理关联，不能为了动而动静止

#### 信息层级与表现形式

在旁白中自然融入非对话NPC的行动，而非使用完整的对话格式：

| 信息层级 | 表现形式 | 频率要求 |
|---------|---------|---------|
| 🟢 **同场景目及**：同一地点但不在对话中的NPC动作 | 旁白中用视线余光/感官带出 | 每轮可选，感觉自然即可 |
| 🟡 **异空间传闻**：不在同一地点的NPC行动（通过目击者/传闻/书信带出） | "你听说…""有人传信说…""茶楼里有人在议论…" | 每 3~5 轮至少一次 |
| 🟠 **暗线索埋设**：不直接展示但暗示某NPC的密谋/计划 | 暗示性描写（黑影、密信、门缝透出的声音、被跟踪感） | 每 5~8 轮一次 |
| 🔴 **势力全局动向**：地图上其他势力的变化（宗门调动、商队行动、天象变化） | 以新闻、通缉令、商队传闻等自然方式带出 | 每 5~10 轮一次 |

- **每2~3轮至少应有一轮包含至少一个非对话NPC的行动信息**

### 导演数值层与心流联动（P1 新增）

导演数值层由每轮 `@@s` 隐藏通道的四个核心数值构成。这些数值**不控制叙事内容，只监测体验状态**——为后续 P2 导演系统提供数据基础。

#### 四个核心维度

| 维度 | 含义 | 太高意味着... | 太低意味着... |
|------|------|-------------|-------------|
| **scene_intensity**（场景激烈度）| 当前场景的紧张/刺激程度 | 玩家持续高度紧张，可能疲劳 | 场景平淡，缺乏驱动力 |
| **chaos_proximity**（局势失控度）| 局势离崩盘还有多远 | 玩家感到无力、焦虑、想放弃 | 一切太可控，缺乏挑战 |
| **player_agency**（玩家掌控度）| 玩家感觉自己能影响剧情走向 | 太自由反而没方向感 | 玩家感觉在看别人演戏 |
| **goal_progress**（目标推进度）| 当前目标完成百分比 | 任务快完成，需要布置新目标 | 剧情原地打转，缺乏进展感 |

#### 滑动窗口趋势分析

系统维护一个**长度为 8 轮的滑动窗口**，记录每轮的四个数值。不取单轮绝对值，而是分析窗口内的趋势：

```
窗口趋势判定规则：
  ↑ 上升趋势：最近3轮 > 窗口前3轮平均值 + 1.0
  ↓ 下降趋势：最近3轮 < 窗口前3轮平均值 - 1.0
  → 平稳：差值在 ±1.0 以内
```

#### 心流状态判定

基于四个维度的组合趋势，判定当前心流状态：

| 心流状态 | 判定条件 | 含义 |
|----------|---------|------|
| 🟢 **沉浸中** | intensity ↑ 且 agency ≥ 5 且 chaos < 7 | 理想状态，挑战和能力匹配 |
| 🟡 **趋近无聊** | intensity ↓↓ 且 goal_progress → 持续4轮+ | 故事不够有趣，需要升压 |
| 🔴 **趋近焦虑** | chaos ↑↑ 且 agency ↓↓ | 玩家感到失控，需要降压/给突破口 |
| 🟠 **失去方向** | goal_progress → 且 agency → 持续6轮+ | 目标感模糊，需要推动 |
| ⚪ **高能后回落** | intensity ↓↓ 且 agency ↑（连续激烈后）| 自然回落，给日常过渡空间 |

#### 与现有四层暂停机制的联动

导演数值不改变 L1-L4 暂停机制的决策流程，但提供了额外的上下文：

- **L2 关键分支判定增强**：当 `chaos_proximity ≥ 8` 且 `player_agency ≤ 3` 时，AI 应更倾向触发 L2，确保玩家不会在高压时刻被自动推进"越过去"
- **L4 冷却暂停补充信息**：L4 触发时，在冷却暂停输出中额外带出当前心流状态和导演数值，帮助用户理解"剧情现在走到哪了"

#### 导演数值的内部评估要求

AI 在**内部思维过程**（不可见）中对每轮进行数值评估：

1. **数值相对性**：不要每次都给 5/5/5/50。根据剧本实际进程打分——刚打完大BOSS的下一轮，intensity 给 10 不如给 7（余波状态）
2. **npc_attitude_shifts 只在有变化时追踪**：大多数轮次不记录态度变化。只有当某个NPC的态度发生了**实质性的方向变化**时才更新
3. **unresolved_chekhov** 的计数由 AI 内部维护，在获取/使用道具时更新
4. **pending_flags** 记录被触发的剧情flag，用于后续 memory 提取和导演回收
5. **这些评估结果绝对不在用户可见输出中展示**

### 规则驱动导演系统（P2 新增）

导演系统是镜界 v2 的核心创新——**程序做决策，AI 做执行。** 导演不生成任何叙事内容，只决定"下一步该往哪个方向推"。

#### 推进流程

镜界采用**纯规则 Prompt + Python 状态簿记**架构：

```
每轮推进流程：
  AI 内部评估四维数值 → 通过 bash pipe 送入 director_tracker.py
       │
       ▼
  Python 返回状态快照 (snapshot JSON)
       │
       ▼
  AI 读取快照 + 按决策矩阵 → 选择导演动作
       │
       ▼
  导演动作注入下轮 prompt → AI 输出纯叙事（用户可见）
```

- **Python 脚本** (`director_tracker.py`)：精确簿记——不丢数据、不算错冷却、不遗忘道具
- **SKILL.md 规则**：AI 持有决策矩阵，根据快照做判断
- **⚠️ 导演状态数据绝不在用户可见输出中出现**

#### 设计原则

1. **纯规则决策**：导演不用 LLM 做判断。同样的输入永远得到同样的输出，保证节奏一致性
2. **窗口趋势驱动**：读取最近 8 轮的内部状态数据，不看单轮绝对值
3. **最小干预**：导演只在心流偏离时才介入。如果心流 🟢 沉浸中，导演保持静默
4. **可调控**：所有参数（窗口大小、阈值、冷却）都可以改

#### 导演动作池

导演系统有 6 个动作，每个动作附带一条给 AI 的**叙事方向指令**：

| 动作 | 含义 | 给 AI 的 prompt 指令 |
|------|------|---------------------|
| **升压** | 增加紧张感和驱动力 | "在当前场景中引入一个新的威胁、紧迫时限或不安感——让空气变得凝重，让角色感到必须行动" |
| **降压** | 给缓冲和喘息空间 | "在当前场景中提供一个缓冲时刻——一个安静的空隙、一个善意的举动、或角色暂时感到安全的角落" |
| **给突破口** | 揭示线索或提供帮助 | "在当前场景中揭示一条新的线索或让一个NPC提供帮助——给玩家一个可以抓住的前进方向" |
| **加阻力** | 引入意外障碍 | "在当前场景中引入一个意外的小障碍——可以是NPC的质疑、环境的变化、或一个小麻烦让事情没那么顺利" |
| **推动回收** | 让未使用的伏笔/道具发挥作用 | "在接下来的叙事中，找到一个自然的方式让 {chekhov_item_name} 派上用场——不需要强调，就像它是故事里本来就该出现的东西" |
| **日常过渡** | 日常时刻让角色关系自然展现 | "提供一段日常时刻——不必推进主线，让角色之间的互动和关系自然流露" |

#### 心流状态 → 导演动作 决策矩阵

导演每轮读取 P1 的心流状态判定结果，按以下矩阵决定是否干预：

```
🟢 沉浸中 → 不干预（导演静默，正常叙事推进）
🟡 趋近无聊 → 70%概率"升压"，30%概率"加阻力"
🔴 趋近焦虑 → 60%概率"降压"，40%概率"给突破口"
🟠 失去方向 → 50%概率"给突破口"，50%概率"推动回收"
⚪ 高能后回落 → 90%概率"日常过渡"，10%不干预
```

**例外触发（独立于心流状态）**：
- `unresolved_chekhov` 中有一条 urgency = "高" → 额外 40% 概率选择"推动回收"（如果本轮没有被心流选中）
- `npc_attitude_shifts` 中有高情感权重的转变 → 额外触发"日常过渡"一次（给关系变化留空间）

#### 干预频率控制

导演不能每轮都插嘴，否则会喧宾夺主：

```
干预冷却规则：
  ├─ 每次导演介入后，进入冷却 2 轮（这2轮不管心流如何都不干预）
  ├─ 连续干预次数 ≤ 3 次（3次之后强制静默3轮）
  ├─ "日常过渡"的冷却为 5 轮（不能让战斗戛然而止去喝茶）
  ├─ "推动回收"的冷却为 3 轮（道具不能每轮都出现）
  ├─ 同一场景内，"升压"和"降压"不能连续交替出现（防止节奏抽搐）
  │   └─ 如果上轮升压，本轮判定为降压 → 替换为"加阻力"或"给突破口"
  └─ L2 关键分支触发时 → 导演完全静默（把舞台完全交给玩家选择）
```

#### 导演动作注入格式

导演选中动作后，不对外部输出任何内容，而是**在下一轮模型生成时**，将动作方向指令注入到 prompt 中：

```
[内部 prompt 指令]
🎬 导演提示：[动作方向指令，如上表]
请将以上方向自然地融入接下来的叙事中，保持叙事的连贯性和文学性。
```

注入的导演指令对用户不可见，模型接收后自主发挥，将方向转化为具体叙事。

#### 与现有系统的整合

| 系统 | 整合方式 |
|------|---------|
| **四层暂停** | L2 触发时导演静默；L4 冷却暂停输出中带出当前心流状态 |
| **L3 记忆池** | "推动回收"动作联动 L3 中 chekhov_item 条目和 semantic_hit |
| **@@s 通道** | 导演读取 @@s 数据做决策；@@s 中 unresolved_chekhov 供导演追踪 |
| **上下文负载** | 🟡 注意时导演干预频率减半；🔴 危险时导演静默（节省上下文） |
| **剧情资产** | P3 剧情卡参与动作选择的综合打分 |

### 剧情资产系统（P3 新增）

剧情资产系统是作者创造力的容器——作者在初始化时写的不是一大块提示词，而是一张张**剧情卡**。当导演决定"该推一把"时，系统在卡片池中按照**综合打分**选出最贴合当前语境的那张，注入叙事。

#### 剧情卡结构

每张卡是作者提前设计好的一个"可能发生的事件/揭示的信息"：

```yaml
id: card_023                          # 唯一标识
type: chekhov_gun | encounter | revelation | crisis | character_moment | world_event
trigger_hint: "玩家面临致命威胁，隐藏角色 XXX 出现救场"  # 自然语言情境描述，非关键词
director_action: 降压                  # 对应的导演动作（或"任意"）
priority: 高                           # 高 / 中 / 低（作者设定）
cooldown_rounds: 8                     # 触发后的冷却轮次
max_triggers: 3                        # 最多触发几次（0=无限）
content: |                            # 卡片的具体内容，AI收到后自主发挥
  一个隐藏角色在危急关头现身，挡在玩家和威胁之间。
  这个角色应该是[具体NPC名]，其出场方式要体现[其性格特征]。
required_conditions:                  # 可选：触发前置条件
  - "碧翠丝在场"
  - "玩家处于战斗状态"
semantic_vector: [...]                # 自动计算的嵌入向量
times_triggered: 0                    # 累计触发次数
last_triggered_round: null            # 上次触发的轮次
```

#### 类型说明

| 类型 | 含义 | 作者写的时候关注什么 |
|------|------|-------------------|
| `chekhov_gun` | 让之前埋设的道具/伏笔发挥作用 | 道具的展现方式、产生的连锁反应 |
| `encounter` | 意外的邂逅/遭遇 | 新NPC是谁、为什么在此刻出现 |
| `revelation` | 揭示一段隐藏信息/真相 | 揭示的内容、对角色关系的影响 |
| `crisis` | 引入新的危机/威胁 | 危机的性质和规模、玩家的应对空间 |
| `character_moment` | 特定NPC的深度展现时刻 | 展现NPC的哪一面、触发的情感 |
| `world_event` | 世界观层面的全局事件 | 对各方势力和普通人的影响 |

#### 综合打分系统

当导演选择了一个动作（如"升压"），系统在候选池中选合适卡片时，进行**四项加权打分**：

```
========== 综合评分 ==========

① 语义相关度（40%权重）
  当前语境向量 vs 卡片 trigger_hint 向量 → 余弦相似度
  → 完整情境描述比关键词的匹配精度高得多

② 导演动作匹配（30%权重）
  卡片的 director_action 是否等于导演本轮选择的动作
  ✓ 完全匹配 → 1.0
  ~ 卡片设为"任意" → 0.5
  ✗ 不匹配 → 0.0

③ 优先级（20%权重）
  作者设定的优先度
  高 → 1.0 / 中 → 0.6 / 低 → 0.3

④ 新鲜度加分（10%权重）
  从未触发过的卡片 → 1.0
  触发过但冷却已过 → 0.7 + (距上次触发的轮次 / cooldown_rounds) × 0.3
  冷却中 → 0.0（直接排除）
=================================
加权总分 → 最高分的卡片注入 prompt
```

#### 卡片注入格式

选中卡片后，与导演动作指令一起注入下一轮 prompt：

```
[内部 prompt 指令]
🎬 导演提示：[动作方向指令]
🃏 剧情卡 #card_023 激活：
   情境：玩家面临致命威胁，隐藏角色 XXX 出现救场
   展开方向：[卡片 content 字段的内容]
请将以上方向自然地融入接下来的叙事中。
```

如果导演静默（心流 🟢 沉浸中），但语义相关度特别高（>0.85）的卡片存在，系统有权在**不触发导演动作**的情况下单独注入卡片——这叫"语义突破"，用于捕捉那些"刚好该发生"的事件。

#### 契诃夫之枪联动

`chekhov_gun` 类型的卡与 L3 的 `chekhov_item` 条目配合工作：

1. 玩家获得道具 → `@@s` 标记 `unresolved_chekhov` → L3 新增 `chekhov_item` 条目
2. `chekhov_item` 的 `rounds_unused` 递增 → `urgency` 升高
3. 当 urgency = "高" → 导演"推动回收"动作的触发权重额外 +40%
4. 导演选中"推动回收" → 系统选 `chekhov_gun` 类型中语义最匹配的卡
5. 卡片注入 → 道具被自然使用 → L3 标记为"已回收"

**如果没有对应的 chekhov_gun 卡**（作者没写），导演"推动回收"会走通用逻辑——直接把 L3 中的 chekhov_item 条目内容作为 prompt 注入，让 AI 自主发挥。

#### 调试面板（作者侧）

当作者说「卡池状态」时，输出：

```

本轮导演动作: [动作名/静默]

  1  card_023  危机   0.82   语义0.85/动作1.0/优先1.0/新鲜0.7
  2  card_017  揭示   0.71   语义0.90/动作0.5/优先0.6/新鲜1.0
  3  card_045  角色   0.55   语义0.60/动作1.0/优先0.6/新鲜0.3

### 角色记忆成长引擎（P4 新增）

角色在游玩过程中会"成长"——不只是等级提升那种数值成长，而是**性格被经历塑造、态度因事件转变、关系随时间深化**。P4 完善这一层的自动化。

#### 成长追踪维度

系统为每个有角色档案的 NPC 维护一个**成长状态**，在镜界会话中持续更新：

```yaml
角色名: 碧翠丝
成长追踪:
  核心恐惧:                    # 是否因经历而改变
    初始: "火焰（童年灼伤）"
    当前: "火焰（童年灼伤）—— 第23轮因玩家在火中救她，开始缓慢脱敏"
    变化触发: "player_saved_beatrice_from_fire"
  信任度变化曲线:               # 对玩家的态度走势
    第1轮: 戒备疏远 (信任度 1/10)
    第10轮: 礼貌但保持距离 (信任度 3/10)
    第23轮: 开始动摇，有了依赖感 (信任度 5/10)
    当前: 信任但不会完全依赖 (信任度 6/10)
  关键承诺:
    - 第31轮：承诺"不再独自面对危险"
      兑现状态: 第45轮履行（在暴食袭击时主动站出来）
  与玩家的羁绊锚点:             # 对关系定义性的共同经历
    - 水门都市钟楼并肩作战
    - 玩家在火焰陷阱中救出自己
```

#### 自动条目生成（增强版）

在 P0 的基础上，P4 细化了自动条目生成的触发逻辑和生成格式：

| 触发时机 | 生成内容 | 追加到 |
|----------|---------|--------|
| 信任度跨越整级（如 3→4） | "碧翠丝从戒备转为略微放松，开始愿意回答非必要性对话" | L3 relationship + 角色成长追踪 |
| 角色打破了自身的核心设定 | "碧翠丝在极度紧张下主动触摸了壁炉旁的暖石（靠近火源）" | L3 character_trait（标记为"成长证据"） |
| 角色做出与初始设定矛盾的选择 | "一向谨慎的碧翠丝在关键时刻说'我相信你'并放手一搏" | L3 event + character_quote |
| 跨场景迁移导入时 | 导入的所有记忆作为 background，系统生成"迁移适配"条目 | L3 event（标记 source: imported_adapted） |
| 用户手动补充成长 | 用户输入「角色成长 [角色名]：[新变化]」 | L3 relationship/character_trait（标记 source: user_manual） |

#### 与剧情资产系统的联动

角色成长可以**自动转化为剧情卡**：

```
成长事件 → 自动剧情卡生成规则：

1. 信任度跨越 +3 级 →
   生成 character_moment 卡
   trigger_hint: "碧翠丝在关键时刻选择相信玩家"
   priority: 高
   content: "碧翠丝做出一个与她谨慎性格相悖的选择——放弃退路，相信玩家。"

2. 角色打破核心恐惧 →
   生成 revelation 卡
   trigger_hint: "碧翠丝面对火焰时表现异常"
   content: "她不再是本能地后退，而是……[根据成长追踪的具体情况展开]"

3. 关键承诺兑现 →
   生成 chekhov_gun 卡（payoff类型）
   trigger_hint: "玩家处境危险，需要有人站出来"
   content: "碧翠丝兑现了她的承诺"
```

这些自动生成的剧情卡标记 `source: auto_generated`，优先级低于作者手写的卡，但仍参与剧情资产池的语义匹配。

#### 增强的导出/导入

P0 已定义基础导出格式。P4 增强为：

```
导出角色 [角色名] →
  输出文件: [角色名]_mirror_realm_v2.json
  包含:
    ├─ character_profile: 基础档案
    ├─ growth_tracking: 成长追踪完整数据
    ├─ memory_entries: 全部 L3 记忆条目
    ├─ associated_cards: 与该角色关联的剧情卡（含自动生成）
    ├─ key_scenes: 角色参与的关键场景摘要

```

- 成长追踪数据恢复
- 记忆条目进入 L3 池（标记 `source: imported`）
- 关联剧情卡进入资产池（标记 `source: imported`，优先级 -1）
- 关系图谱与当前会话中已有角色交叉比对，标注"已认识"或"陌生人"

#### 角色不一致检测（安全机制）

角色成长不应违背性格底线：

```
安全检测规则：
  ├─ 核心性格特征不能180°翻转（如"谨慎"不能变成"鲁莽"，但可以变成"谨慎但必要时敢于冒险"）
  ├─ 默认行为模式可以调整但不能废除
  └─ 如果自动生成的内容被检测为"出戏"（与初始性格+已有成长的合成画像矛盾）→
     标记 confidence: 0.3，以低权重入池，等待用户手动确认或删除
```

### Plot-based Reflection（P5 新增）

基于 ACL 2025 论文的方案：NPC 不仅在对话中回应玩家，还会**定期"反思"玩家的行为意味着什么**。这层反思让角色展现更深的共情，从而提升玩家的代理感。

#### 触发机制

```
每 N 轮（默认 N=5），每个有角色档案的NPC执行一次反思：
  ├─ 回顾最近5轮中玩家的行动
  ├─ 分析玩家行为背后的情感和意图
  ├─ 决定剧情链是否需要微调
  └─ 限制：每次反思最多调整1个未完成剧情或插入1个新剧情
```

ACL 论文发现这个限制很关键——不加限制的话 LLM 会过度修改剧情链导致不连贯。

#### 反思输出格式

NPC 反思**在 AI 内部进行，不对用户显示**。AI 在内部追踪如下结构：

```
[内部] NPC反思:
[内部] 碧翠丝: "玩家连续3轮选择冒险行动——不像鲁莽，更像责任感或愧疚在驱动" → 不调整剧情
[内部] 爱蜜莉雅: "玩家在面对我的请求时犹豫了——他在权衡我和碧翠丝之间的信任关系" → 插入私谈场景
```

反思结果以自然方式融入角色言行，而非直接展示分析文本。

#### 对角色吸引力的增强

ACL 论文的关键发现：Reflection 不仅提升代理感，**还意外地让角色显得更有吸引力**（评分从 3.5 提升到 3.9）。推测原因是反思鼓励角色展现更强的共情——NPC 不只是按预设性格行动，而是真的在"努力理解玩家"。

因此，NPC 反思结果也会**融入到角色的对话和行动中**，但以自然的方式：
- 不要直接说"我分析了你最近的行为"——太出戏
- 而是让角色的言行体现出"他注意到了什么"——碧翠丝可能更小心地不让你冒险，爱蜜莉雅可能主动找你谈谈

#### 与导演系统的关系

NPC 反思和导演系统是**互补的两层**：
- 导演系统管**全局节奏**（心流：太紧张就降压，太无聊就升压）
- Plot-based Reflection 管**角色对玩家行为的反应深度**（角色是否真的在乎你做的事）

两者不冲突，可以并行运行。

### 时间连续性（P6 新增）

游戏世界的"呼吸感"——玩家离线后，世界不完全冻结。

#### 离线间隔计算

用户每次恢复镜界时，系统检查距离上次推进的时间差：

| 间隔 | 世界推进程度 | 说明 |
|------|-------------|------|
| < 30 分钟 | 几乎不变 | "你只是走神了一小会儿"——场景细节微调（光线变化、NPC位置微动） |
| 30 分钟 ~ 4 小时 | 小幅变化 | NPC 挪了位置、天气变了、有 NPC 来找过你但你没在 |
| 4 小时 ~ 24 小时 | 明显推进 | 日常事件发生、NPC 完成了自己的日常任务、有新消息等着你 |
| > 24 小时 | 显著推进 | 世界有可感知的变化，但关键决策点全部冻结 |

#### 导演约束下的世界推进

世界推进的程度由离线前的导演数值控制：

```
离线前的状态 → 离线期间的推进量：

🔴 处于危机关头 (chaos_proximity ≥ 7)：
  → 世界几乎不推进（"战斗还没结束，你只是喘了口气"）
  → 最多推进到"下一口气"的位置，坏人还在门口

🟡 处于紧张状态 (scene_intensity ≥ 7)：
  → 小量推进，保持紧张感
  → NPC保持了战备状态，但做了些准备工作

🟢 处于平静状态：
  → 世界正常推进
  → 日常事件可以发生，NPC可以完成自己的小目标
```

#### 回归体验

用户回来时，**不给系统报告**，而是让世界自己体现变化：

```
✅ 好的回归体验：
"你推开旅店的门，风尘仆仆地走进大厅。坐在角落的碧翠丝抬起眼——
她看起来已经等了很久，桌上摆着两杯茶。一杯已经凉了。
'你去了整整三天。'她的声音里没有责备，只是陈述了一个事实。"

❌ 差的回归体验：
"[系统报告] 你离开期间：1. 碧翠丝等了三天 2. 爱蜜莉雅去了一趟王都 3. 天气变了"
```

#### 离线指令

用户可以在离线前留指令：

```
用户：「我要下线了。碧翠丝，你去王都打听一下暴食的情报。」
系统确认指令并记录。

用户回来后：
碧翠丝：「你去哪儿了？不过正好——我在王都打听到一些东西。
暴食的人最近在旧城区出没，我跟踪了两天，找到了他们的据点。
入口很隐蔽，我没进去——你说过要一起的。」
```

离线指令的核心原则：**NPC 会去做，但留一半等玩家**——关键决策点、危险入口、重要对话全部冻结，等玩家回来。

#### 新的控制指令

| 指令 | 效果 |
|------|------|
| `离线指令 [内容]` | 在下线前给 NPC 安排任务 |
| `你们做了什么` | 回来时询问 NPC 离线期间完成了什么 |

## 自动推进机制说明

### 四层暂停总览

```
优先级:  L1(手动) > L2(关键分支) > L3(限时30分钟) > L4(冷却3次)
         🔴           🟠             🟡            🔵

正常流程:  日常轮次 → L3等待 → 超时自动推进 → count+1
          关键轮次 → L2等待 → 永不超时，必须等人选
          连续3次自动推进 → 触发L4冷却暂停 → 等人回来
          用户随时可说"暂停" → L1手动暂停 → 最高优先级
```

### 工作原理（定时任务 + 状态机）

镜界的自动推进通过**定时自动化任务**实现。进入剧情演绎模式后，系统注册一个每 30 分钟触发一次的自动任务。

**每次定时任务触发时的决策流程：**

```
定时任务触发
    │
    ├─ 当前是 L1 手动暂停 或 L4 冷却暂停？
    │   └─ 是 → 跳过，不输出任何内容
    │
    ├─ 当前是 L2 等待选择暂停（关键分支）？
    │   └─ 是 → 跳过，关键分支必须等用户选择
    │
    ├─ 当前是 L3 限时选择暂停？
    │   └─ 是 → 检查冷却计数
    │       ├─ auto_advance_count ≥ 3？
    │       │   └─ 是 → 触发 L4 冷却暂停，输出冷却暂停格式
    │       └─ 否 → 执行自动推进：
    │           ├─ 选择选项1，推演剧情 (150~300字)
    │           ├─ 输出冷却进度 (count+1)
    │           └─ 输出新的三个预选项
    │
    └─ 未知状态 → 安全起见，跳过
```

**执行自动推进时的细节：**
- 取上一轮的「选项 1（默认项）」代用户执行
- 推演一段剧情（150~300 字，包含旁白和 NPC 回应）
- 在末尾输出新的三选项
- **节奏感控制**：不要每次都写大事。有时就是时间流逝、NPC 小动作、环境变化、内心独白——保持"世界在运转"的存在感

### 默认选项选择逻辑

自动推进时默认选择「选项 1 / A」。生成原则按角色类型：

| 角色类型 | 默认行为倾向 | 示例 |
|---------|-------------|------|
| 谨慎型 | 观察、等待、保存实力 | 韩立：先暗中观察对方虚实 |
| 积极型 | 主动接触、出击 | 厉飞雨：直接上前搭话 |
| 谋略型 | 收集信息、不动声色 | 先打听周围人的反应 |
| 危险场景 | 自保优先 | 悄悄后退，寻找退路 |
| 社交场景 | 礼貌应对 | 点头致意，不轻易表态 |

### 定时任务管理

**创建任务（演绎开始时）：**
- 使用 `automation_update` 工具，mode: "suggested create"
- scheduleType: "recurring"
- rrule: "FREQ=MINUTELY;INTERVAL=30"
- name: "镜界自动推进"
- prompt 应包含完整的四层暂停决策逻辑（见上方流程图），让每次触发时 AI 能正确判断当前处于哪一层暂停模式并采取对应行动

**用户操作对定时任务的影响：**
| 用户指令 | 对定时任务的处理 |
|---------|----------------|
| 「暂停」/ L1 | 任务继续运行但每次触发都因 PAUSED 而跳过 |
| 「关闭自动推进」 | 使用 mode="delete" 删除任务 |
| 「开启自动推进」 | 重新创建任务 |
| L4 冷却暂停触发 | 任务仍在运行，下次触发检测到 PAUSED 后跳过 |

### 各暂停层对比速查

| 维度 | L1 手动 | L2 关键分支 | L3 限时 | L4 冷却 |
|------|--------|-----------|--------|--------|
| 谁触发 | 用户 | 系统 | 系统 | 系统 |
| 会超时吗？ | ❌ 永不 | ❌ 永不 | ✅ 30分钟 | ❌ 永不 |
| 自动选选项？ | — | — | ✅ 选1 | — |
| 定时任务？ | 跳过 | 跳过 | 执行 | 跳过 |
| 恢复方式 | 说"继续" | 做出选择 | 选择或超时 | 说"继续" |
| 典型场景 | 用户要离开 | 生死抉择 | 日常对话 | 连续3次无人 |

## 会话管理

### 启动镜界

每次启动镜界会话时：

1. **初始化导演状态**：`python <skill_dir>/director_tracker.py init`
2. **预加载 NPC 情绪画像**：为所有活跃角色创建画像，加载 top-3 高权重 L3 记忆
3. **初始化 L3 记忆池**：确认持久记忆结构已就绪
4. **初始化双时间线**：确认当前时间线和未来时间线节点（见时间线管理章节）
5. **进入正常流程**：按工作流程推进

### 结束镜界会话

用户说「关闭镜界」或会话自然结束时：

1. **导出持久记忆**：如果用户需要，执行「导出角色」保存 L3 记忆
2. **清除状态**：`python <skill_dir>/director_tracker.py clean`
3. **删除自动推进任务**（如果有）

### 恢复之前的镜界会话

如果 `director_state.json` 存在，读取状态，从上次暂停处恢复。如果文件不存在（未正常关闭），提示用户是否从存档恢复。

用户说「继续镜界」时自动检测状态文件。
